## 1. Контроллер домена Samba на BR-SRV

Домен au-team.irpo. На ALT ставится метапакет task-samba-dc, обычный samba не содержит samba-tool.

```bash
apt-get install task-samba-dc -y
systemctl stop smb nmb 2>/dev/null
mv /etc/samba/smb.conf /etc/samba/smb.conf.bak 2>/dev/null

samba-tool domain provision \
  --realm=AU-TEAM.IRPO \
  --domain=AU-TEAM \
  --server-role=dc \
  --dns-backend=SAMBA_INTERNAL \
  --use-rfc2307 \
  --adminpass='P@ssw0rd'

ln -sf /var/lib/samba/private/krb5.conf /etc/krb5.conf
systemctl disable --now bind 2>/dev/null
systemctl enable --now samba
```

BR-SRV должен резолвить через себя:

```bash
echo -e 'search au-team.irpo\nnameserver 127.0.0.1' > /etc/resolv.conf
samba-tool domain info 127.0.0.1   # проверка
```

Группа hq и 5 пользователей:

```bash
samba-tool group add hq
for i in 1 2 3 4 5; do
  samba-tool user create hquser$i 'P@ssw0rd'
  samba-tool group addmembers hq hquser$i
done
samba-tool user list
```

Sudo для группы hq только cat, grep, id. Файл создаётся на HQ-CLI (там пользователи входят):

```bash
echo '%hq ALL=(ALL) NOPASSWD: /bin/cat, /bin/grep, /usr/bin/id' > /etc/sudoers.d/hq
chmod 440 /etc/sudoers.d/hq
```

Если sssd использует полные имена, группа пишется как `%hq@au-team.irpo`. Проверь через `id hquser1@au-team.irpo` после ввода в домен.

### Ввод HQ-CLI в домен

На HQ-CLI DNS должен указывать на BR-SRV:

```bash
hostnamectl set-hostname hq-cli.au-team.irpo
echo -e 'search au-team.irpo\nnameserver <IP_BR-SRV>' > /etc/resolv.conf

apt-get install task-auth-ad-sssd -y
system-auth write ad au-team.irpo hq-cli AU-TEAM administrator
# введёт пароль администратора домена P@ssw0rd
```

Проверка ввода и прав на аутентификацию:

```bash
getent passwd hquser1@au-team.irpo
# вход под доменным пользователем должен работать
```

Если нужно ограничить вход только группой hq, в `/etc/sssd/sssd.conf` добавь `access_provider = simple` и `simple_allow_groups = hq`, затем `systemctl restart sssd`.

## 2. RAID на HQ-SRV

Уровень массива, имя устройства и число дисков зависят от варианта. Точные значения в [[Варианты]]. Точка монтирования у всех /raid, файловая система ext4.

```bash
apt-get install mdadm -y

# Пример RAID 5 (В2, В3, В4). Для В1 это RAID 1 на двух дисках. Имя mdX по варианту.
mdadm --create /dev/mdX --level=5 --raid-devices=3 /dev/sdb /dev/sdc /dev/sdd
cat /proc/mdstat        # дождаться сборки

mkfs.ext4 /dev/mdX
mkdir /raid
mount /dev/mdX /raid

mdadm --detail --scan >> /etc/mdadm.conf
echo '/dev/mdX  /raid  ext4  defaults  0 0' >> /etc/fstab
mount -a
```

Имена дисков уточни через `lsblk`. Для RAID 1 ставь `--level=1 --raid-devices=2`.

## 3. NFS на HQ-SRV

Папка общего доступа /raid/nfs, чтение и запись только для сети в сторону HQ-CLI.

```bash
apt-get install nfs-server nfs-utils -y
mkdir -p /raid/nfs
```

`/etc/exports` (подсеть HQ-CLI с маской /28 из своего варианта):

```
/raid/nfs  <сеть_HQ-CLI>/28(rw,sync,no_root_squash,subtree_check)
```

```bash
systemctl enable --now nfs
exportfs -ra
exportfs -v        # проверка
```

### HQ-CLI, автомонтирование в /mnt/nfs

```bash
mkdir -p /mnt/nfs
```

`/etc/fstab`:

```
hq-srv.au-team.irpo:/raid/nfs  /mnt/nfs  nfs  defaults,x-systemd.automount  0 0
```

```bash
systemctl daemon-reload
mount -a
df -h | grep nfs
```

Опция x-systemd.automount даёт монтирование по обращению. Без неё убери её, тогда смонтируется при загрузке.

## 4. Chrony на ISP

Сервер времени на ISP, стратум по варианту (см. [[Варианты]]). Клиенты HQ-SRV, HQ-CLI, BR-RTR, BR-SRV.

ISP, `/etc/chrony.conf`:

```
pool pool.ntp.org iburst
local stratum <6|7|8|9>
allow 0.0.0.0/0
```

```bash
systemctl enable --now chronyd
```

Клиенты, в `/etc/chrony.conf` убери pool и добавь:

```
server <адрес ISP> iburst
```

HQ-SRV и HQ-CLI указывают на интерфейс ISP в сторону HQ, BR-RTR и BR-SRV на интерфейс ISP в сторону BR (адреса по варианту в [[Варианты]]).

```bash
systemctl enable --now chronyd
chronyc sources -v
chronyc tracking
```

## 5. Ansible на BR-SRV

Инвентарь с HQ-SRV, HQ-CLI, HQ-RTR, BR-RTR. Рабочий каталог /etc/ansible. Цель: чистый pong без ошибок и предупреждений.

```bash
apt-get install ansible -y
mkdir -p /etc/ansible
```

`/etc/ansible/ansible.cfg`:

```ini
[defaults]
inventory = /etc/ansible/hosts
host_key_checking = False
interpreter_python = /usr/bin/python3
```

`/etc/ansible/hosts` (порт SSH серверов по варианту, маршрутизаторы обычно 22):

```ini
[servers]
hq-srv.au-team.irpo ansible_port=<2011..2014> ansible_user=sshuser
hq-cli.au-team.irpo
br-rtr.au-team.irpo ansible_user=net_admin

[routers]
hq-rtr.au-team.irpo ansible_user=net_admin

[all:vars]
ansible_python_interpreter=/usr/bin/python3
```

Раздай ключи, чтобы не было запросов пароля и предупреждений:

```bash
ssh-keygen -t ed25519 -N '' -f ~/.ssh/id_ed25519
ssh-copy-id -p <порт> sshuser@hq-srv.au-team.irpo
ssh-copy-id net_admin@hq-rtr.au-team.irpo
ssh-copy-id net_admin@br-rtr.au-team.irpo
ssh-copy-id <user>@hq-cli.au-team.irpo

ansible all -m ping
```

Все хосты должны вернуть `"ping": "pong"` со статусом SUCCESS. Предупреждение про интерпретатор Python убирается переменной interpreter_python.

## 6. Docker-приложение на BR-SRV

Стек из двух контейнеров: site и db. Образы site_latest и mariadb_latest или postgresql_latest лежат в каталоге docker образа Additional.iso. Имя БД, пользователь, пароль, порт и образ БД зависят от варианта (см. [[Варианты]]).

```bash
apt-get install docker-ce docker-compose -y
systemctl enable --now docker

# Подключить Additional.iso и загрузить образы
mkdir -p /mnt/iso && mount /dev/sr1 /mnt/iso 2>/dev/null
docker load -i /mnt/iso/docker/site_latest.tar
docker load -i /mnt/iso/docker/<mariadb|postgresql>_latest.tar
docker images
```

Файл `~/app.yml` бери из [[Варианты]] под свой номер. Запуск:

```bash
docker compose -f ~/app.yml up -d
docker ps        # site и db в статусе Up
```

Приложение должно отвечать на внешнем порту своего варианта. Проброс этого порта снаружи делается на BR-RTR в разделе 8.

## 7. Веб-приложение на HQ-SRV

Apache, MariaDB, импорт dump.sql в базу webdb. Пользователь webN по варианту. Файлы index.php и images из каталога web образа Additional.iso.

```bash
apt-get install apache2 mariadb-server php8.2 php8.2-mysqli -y
systemctl enable --now mariadb
systemctl enable --now httpd2
```

База и пользователь (webN из [[Варианты]]):

```bash
mysql -u root <<'SQL'
CREATE DATABASE IF NOT EXISTS webdb;
CREATE USER 'webN'@'localhost' IDENTIFIED BY 'P@ssw0rd';
GRANT ALL PRIVILEGES ON webdb.* TO 'webN'@'localhost';
FLUSH PRIVILEGES;
SQL

mysql -u root webdb < /mnt/iso/web/dump.sql
```

Файлы приложения:

```bash
cp /mnt/iso/web/index.php /var/www/html/
cp -r /mnt/iso/web/images /var/www/html/
```

В `/var/www/html/index.php` пропиши учётные данные подключения: пользователь webN, пароль P@ssw0rd, база webdb, хост localhost.

```bash
systemctl restart httpd2
curl -s localhost | head    # проверка
```

Корень Apache на ALT это /var/www/html, служба называется httpd2.

## 8. Проброс портов на маршрутизаторах

Статическая трансляция портов (DNAT) на HQ-RTR и BR-RTR.

Кратко, что пробрасывается:
- HQ-RTR: внешний порт веб-приложения в HQ-SRV, порт 20XX в SSH HQ-SRV
- BR-RTR: внешний порт docker-приложения в BR-SRV, порт 20XX в SSH BR-SRV

Внимание по В2: SSH на BR-RTR пробрасывается в порт 2026, а не 2012.

```bash
# HQ-RTR
WEBPORT=8081  # 8081/8082/8083/8084
SSHPORT=2011  # 2011/2012/2013/2014
iptables -t nat -A PREROUTING -i enp0s3 -p tcp --dport $WEBPORT -j DNAT --to-destination 192.168.10.2:$WEBPORT
iptables -t nat -A PREROUTING -i enp0s3 -p tcp --dport $SSHPORT -j DNAT --to-destination 192.168.10.2:$SSHPORT
iptables -A FORWARD -p tcp -d 192.168.10.2 --dport $WEBPORT -j ACCEPT
iptables -A FORWARD -p tcp -d 192.168.10.2 --dport $SSHPORT -j ACCEPT
iptables-save > /etc/sysconfig/iptables

# BR-RTR
WEBPORT=8081       # свой вариант
SSHPORT_BR=2011    # В2: 2026, остальные: тот же что HQ
iptables -t nat -A PREROUTING -i enp0s3 -p tcp --dport $WEBPORT -j DNAT --to-destination 192.168.30.2:$WEBPORT
iptables -t nat -A PREROUTING -i enp0s3 -p tcp --dport $SSHPORT_BR -j DNAT --to-destination 192.168.30.2:$SSHPORT_BR
iptables -A FORWARD -p tcp -d 192.168.30.2 --dport $WEBPORT -j ACCEPT
iptables -A FORWARD -p tcp -d 192.168.30.2 --dport $SSHPORT_BR -j ACCEPT
iptables-save > /etc/sysconfig/iptables
```

## 9. Nginx обратный прокси и web-аутентификация на ISP

На ISP nginx проксирует по доменным именам. web.au-team.irpo ведёт на веб-приложение HQ-SRV, docker.au-team.irpo ведёт на контейнер site BR-SRV. На web.au-team.irpo включается basic-аутентификация, логин по варианту (см. [[Варианты#nginx basic auth]]), пароль P@ssw0rd.

```bash
apt-get install nginx apache2-utils -y
htpasswd -cb /etc/nginx/.htpasswd <логин_варианта> P@ssw0rd
```

`/etc/nginx/sites-available.d/proxy.conf` (порты и внешние адреса роутеров по варианту):

```nginx
server {
    listen 80;
    server_name web.au-team.irpo;

    auth_basic "Restricted";
    auth_basic_user_file /etc/nginx/.htpasswd;

    location / {
        proxy_pass http://<внешний_адрес_HQ-RTR>:<порт_веб>;
        proxy_set_header Host $host;
    }
}

server {
    listen 80;
    server_name docker.au-team.irpo;

    location / {
        proxy_pass http://<внешний_адрес_BR-RTR>:<порт_docker>;
        proxy_set_header Host $host;
    }
}
```

```bash
ln -s /etc/nginx/sites-available.d/proxy.conf /etc/nginx/sites-enabled.d/ 2>/dev/null
nginx -t && systemctl enable --now nginx
```

Аутентификация только на web.au-team.irpo. На docker.au-team.irpo блок auth_basic не добавляется. Доменные имена docker и web резолвятся в адреса интерфейсов ISP, эти записи преднастроены на стенде.
